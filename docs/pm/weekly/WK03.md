## Blockers encountered

- **MLflow 3.x artifact path change:** In MLflow 3.x, logged models are stored under `mlruns/1/models/m-{uuid}/artifacts/` rather than under run IDs. Loading with `runs:/{run_id}/artifact_name` requires the SQLite tracking database, which stores host-absolute paths unreachable inside Docker. Fix: load models by direct file path using the UUID, which can be queried with `sqlite3 mlflow.db "SELECT model_id, name FROM logged_models;"`.

- **Docker container networking:** Inside the Streamlit container, `localhost:8000` refers to the container itself, not the FastAPI container. Fix: inject `API_URL=http://api:8000` as an environment variable for the Docker service, with `localhost:8000` as the local development fallback.

- **Stale Docker image:** After editing Python source files, `docker compose up` (without `--build`) runs the old cached image. Code changes on disk have no effect until the image is rebuilt.

---

## What I learned

Serving a trained model inside Docker introduces a gap between development and production assumptions that does not exist in notebooks. Paths, hostnames, and environment variables all change. The cleanest pattern is to inject configuration as environment variables and load model artifacts by explicit file path rather than relying on a tracking server that was configured for a different filesystem.
