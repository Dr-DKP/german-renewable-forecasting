# Retrospectives

> **Corporate vocab:** A *retrospective* (or *retro*) is a structured reflection at the end of a phase or sprint. The classic format is *Start / Stop / Continue*: what should we start doing, what should we stop doing, what is working and should continue. Agile teams run retros every 1-2 weeks. For a solo portfolio project, the retro becomes a written artifact that demonstrates *growth mindset under pressure* — recruiters read this to understand how you learn.

## Cadence for P2 (2 retros)

- **Mid-sprint retro** — end of Day 13 (2026-05-01) — covers Phases 0-3
- **Final retro** — end of Day 22 (2026-05-12) — covers the whole sprint

Claude facilitates by asking the Start / Stop / Continue questions at the Day 13 and Day 22 close blocks. User answers; Claude writes up the retro file; user edits and commits.

## Template (copy to `mid-sprint-retro.md` / `final-retro.md`)

```markdown
# Retrospective — <Phase / Sprint Name>

**Date:** YYYY-MM-DD
**Covers:** Days N-M

## Start
Things I should start doing next phase.

## Stop
Things that wasted time or added friction.

## Continue
Things that worked and should keep happening.

## Biggest surprise
The one thing I didn't predict.

## Biggest mistake
The one thing I'd redo. Not punishment — learning.

## What I'd tell a teammate starting this phase tomorrow
(3-5 bullet points)

## Changes to scope / plan from this retro
(link to ADR if scope changed)
```
